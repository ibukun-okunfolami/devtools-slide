Why This Training Matters

Problems today

  • "It's not working" tickets
  • Engineering asks follow-up questions
  • Slow resolution time

What DevTools fixes

  • Clear proof of failure
  • Faster issue isolation
  • Fewer back-and-forth messages

What You Will Learn

By the end of this session, you will be able to:

  • Identify frontend vs backend issues
  • Capture console errors correctly
  • Inspect network requests & responses
  • Submit engineering-ready bug reports

What This Training Is Not

  • You are not becoming an engineer
  • You are not fixing bugs
  • You are collecting evidence engineers need

DevTools Overview

Main tabs we'll use

  • Console → JavaScript errors
  • Elements → UI / visibility issues
  • Network → requests, responses, timing

Opening DevTools

  • Right-click → Inspect

Shortcut:

  • Mac: Cmd + Option + I
  • Windows: Ctrl + Shift + I

Console Tab (Most Important)

What the Console shows:

  • Errors (red)
  • Warnings (yellow)
  • Logs (info)

Golden rule: If it's red, screenshot it.

Common Console Errors CS Should Recognize

  • Uncaught TypeError
  • Failed to fetch
  • CORS error
  • 401 / 403 authorization errors

Elements Tab (UI Issues)

Use Elements to:

  • Check if a button exists
  • See if it's hidden (display: none)
  • See if it's disabled

When to Use Elements

Examples:

  • "Button is missing"
  • "Text is wrong"
  • "Page layout is broken"

Network Tab Mental Model

  • Every user action = a network request
  • If nothing shows → frontend issue
  • If request fails → backend or auth issue

How to Capture Network Requests

Steps:

  1. Open Network tab
  2. Perform the action
  3. Filter by Fetch/XHR

The Network Tab

Yellow: The Network tab is selected. The Fetch/XHR filter (bottom) is active—only Fetch and XHR requests are shown, so you can focus on API calls.

Green: Successful requests (e.g. get-cart-value, get-cart-details) with Status 200 (OK). These are xhr requests; Size and Time show how much data and how long they took.

Red: Failed requests (e.g. get-wallet-balance, get-paylater-balance) with Status 400 (Bad Request). The red X means the server rejected the request—often due to invalid or missing data.

Understanding a Request

What engineers care about:

  • Endpoint (URL)
  • Method (GET / POST)
  • Status code
  • Response message

Important screenshots to take for a failed API request

Status Codes CS Must Know

  • 200 — OK (success)
  • 400 — Bad Request
  • 401 — Unauthorized (login/session)
  • 403 — Forbidden (no permission)
  • 404 — Not Found
  • 500 — Server error

Timing & Performance

Signs of issues:

  • Pending / stalled requests
  • Very slow responses (>5s)

What Engineers Need (Minimum)

A bug report must include:

  • Steps to reproduce
  • Expected vs actual result
  • Console evidence
  • Network evidence
